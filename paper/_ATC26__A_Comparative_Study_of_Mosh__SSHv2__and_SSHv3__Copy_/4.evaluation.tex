\section{Performance Evaluation}
\label{sec:evaluation}
In this section, we conduct extensive experiments to answer the following research question: \textbf{Which SSH Connection Protocol can meet the strict latency and completeness demands of closed-loop terminal agents?}

The setup consists of a benchmark client, a network path, and a remote server. Two Raspberry Pi 4 devices are used as the client and the server, respectively. Because Linux \texttt{netem} \sout{shapes egress traffic} \textcolor{blue}{only applies network attenuation to outgoing traffic}, the same \sout{profile} \textcolor{blue}{scenario} is applied on both hosts to emulate bidirectional degradation. A configured one-way delay $D$ therefore yields approximately $2D$ ms RTT, and a per-egress loss rate $p$ yields approximately $1-(1-p)^2$ round-trip loss exposure. We evaluate concrete protocol implementations rather than abstract protocol models. SSHv2 is implemented with OpenSSH 9.2 and OpenSSL 3.0.17. Mosh is implemented with Mosh 1.4.0 and uses the standard SSH bootstrap procedure before switching to its UDP-based State Synchronization Protocol channel. SSH3 is implemented using the francoismichel/ssh3 implementation and connects to the SSH3 terminal server endpoint \cite{michel2023}. All protocols use the same remote account, SSH key material, server host, terminal environment, and workload content.

For each protocol, workload, and scenario, the benchmark runs 50 independent trials. Each trial establishes a fresh session and then executes 100 commands. Session setup latency is measured once per trial, giving 10 setup measurements for each configuration. Workload latency is measured once per command or probe iteration, giving 1000 workload samples for each configuration. Within each scenario, protocol, workload, and command or probe combinations are shuffled \sout {using a fixed seed} \textcolor{blue}{using a fixed initialization value} to reduce order effects from cache state, device temperature, run order, and background variation.

Session setup latency is measured from client process spawn to the first interactive shell prompt. Figure~\ref{fig:session-setup} shows that SSH3 has the lowest setup latency in every network scenario. \sout{On the VPN path, SSH3 reaches the first usable prompt in 876.0\,ms, compared with 2873.3\,ms for SSHv2 and 3280.0\,ms for Mosh. The same ordering holds under emulated impairment: SSH3 requires 982.3\,ms in Low, 1294.2\,ms in Medium, and 1685.7\,ms in High, while SSHv2 requires 1674.3\,ms, 2577.1\,ms, and 3960.5\,ms, and Mosh requires 2270.5\,ms, 3503.6\,ms, and 5161.6\,ms} \textcolor{blue}{In the VPN scenario, SSH3 took 876.0\,ms to reach the first usable shell prompt, compared with 2873.3\,ms for SSHv2 and 3280.0\,ms for Mosh. The same ordering holds in the controlled LAN scenarios: SSH3 took 982.3\,ms, 1294.2\,ms, and 1685.7\,ms to reach the first usable shell prompt in the Low, Medium, and High scenarios, respectively. This result follows from the connection-establishment path of each protocol. SSH3 runs over HTTP/3 and QUIC \cite{michel2023}, whose handshake combines transport and cryptographic negotiation \cite{Iyengar2021}. SSHv2 first establishes TCP and then performs SSH key exchange and authentication \cite{Ylonen2006RFC4253}. \sout{Mosh pays the highest startup cost because it uses SSHv2 to authenticate and bootstrap the session before switching to its UDP-based State Synchronization Protocol channel \cite{Winstein2012}. For an AI-like terminal client, this cold-start cost matters when the client opens fresh shells, reconnects after failures, or runs short-lived remote-control loops}. \textcolor{blue}{Mosh had the longest session setup time in these measurements because it uses SSHv2 for authentication and session establishment before switching to a UDP-based State Synchronization Protocol (SSP) channel \cite{Winstein2012}. In the context of interactive terminal tasks like AI, this session startup latency can impact measurement results and execution efficiency, especially when the task frequently opens new shells, restores connections, or runs multiple short remote control loops}.

W1 measures simulated interactions on commands with short response times and small output. As shown in Figure~\ref{fig:latency-w1}, SSH3 \sout{achieved} \textcolor{blue}{achieved} the lowest mean W1 latency across four network scenarios, although the margins were small on slightly degraded paths. In a real-world internet environment, SSH3 completed the command loop in 177.9 ms, compared to 188.8 ms for SSHv2 and 190.9 ms for Mosh. In the Low volatility network scenario, SSH3 and SSHv2 were nearly equal at 75.5 ms and 76.1 ms, respectively, while Mosh achieved 82.2 ms. The difference became apparent when network volatility was at High, where SSH3 achieved 307.5 ms, outperforming Mosh at 317.3 ms and SSHv2 at 367.0 ms. This result shows that the QUIC-based terminal channel of SSH3 handles short response loops efficiently. QUIC integrates encryption, encapsulation, acknowledgment, and loss recovery at the transport layer. In SSH3, this gives the tested terminal channel a shorter W1 completion path compared to SSHv2 over TCP and Mosh over SSP. SSHv2 preserves the completeness of the byte stream, but the loss and recovery of the stream are delayed, increasing completion latency in faulty paths. Mosh is optimized for more intuitive responsiveness rather than full byte command loop completion, so it offers less advantage to W1 \sout{as shown in the table}\textcolor{blue}{as reflected by the W1 latency and output-completeness results in Figure~\ref{fig:latency-w1}}.

The main reason is that W2 uses \sout{one terminal session}\textcolor{blue}{terminal sessions} and sends the command output sequentially through \sout{one}\textcolor{blue}{a} shell stream. Therefore, \sout{when}\textcolor{blue}{because} W2 uses only one stream, SSH3 does not benefit from QUIC's \sout{multi-stream recovery behavior}\textcolor{blue}{multi-stream recovery behavior} \cite{Iyengar2021}. W3 evaluates keystroke-to-echo latency to capture client-visible responsiveness during interactive terminal use. Overall, Mosh consistently outperforms the two byte-stream protocols (SSH3 and SSHv2). In the 1p \texttt{tmux} layout in Figure~\ref{fig:latency-w3-1p}, Mosh demonstrates a substantial advantage on impaired paths, maintaining a latency of just 25.3 ms under High impairment. In contrast, both SSHv2 and SSH3 experience severe degradation, with both exceeding 220 ms under the same conditions. This behavioral gap in the 1p scenario stems from fundamental architectural differences. Mosh achieves Low latency through predictive local echo and screen-state synchronization, hiding network delay from client-visible rendering \cite{Winstein2012}. Conversely, both SSH3 and SSHv2 operate strictly as end-to-end byte streams. Regardless of whether they utilize modern QUIC or legacy TCP, both protocols must wait for the remote server to process the keystroke and send the output back, making their responsiveness entirely dependent on network round-trip time and terminal rendering overhead. Moving to the more complex 5p layout in Figure~\ref{fig:latency-w3-5p}, the performance gap between Mosh and the byte-stream protocols widens further. Mosh remains highly responsive, recording only 31.3 ms under High impairment. Meanwhile, the latency for both SSH3 and SSHv2 scales up significantly. This magnified delay occurs because increasing the number of panes increases the volume of screen state updates that the server must push to the client. Since SSH3 and SSHv2 are obligated to transmit every byte of the terminal redrawing sequence, their interactive latency inherently inflates. In contrast, Mosh efficiently coalesces these complex layout updates and predicts visible changes locally, preventing the network from becoming a rendering bottleneck. Consequently, W3 represents a fundamentally different operational regime compared to W1 and W2. While SSH3 and SSHv2 exhibit restrictive, network-bound characteristics during interactive editing, Mosh stands out as the definitively superior choice for masking network impairments and preserving human-driven responsiveness.


Returning to our central research question, our evaluation reveals that no single transport universally satisfies all phases of agent interaction. Instead, the optimal protocol is workload-dependent. For agile, short-lived interactions requiring rapid session establishment and Low command-response latency, SSH3’s QUIC-based transport provides the most efficient feedback channel. However, for programmatic observation phases that rely on parsing large terminal outputs, legacy SSHv2 remains the superior choice, delivering the fastest byte-complete transmission despite its TCP roots. Crucially, our results demonstrate that transport mechanisms optimized for human visual responsiveness, such as Mosh, inherently conflict with the programmatic needs of automated clients, sacrificing critical byte-completeness in heavy-output scenarios.
