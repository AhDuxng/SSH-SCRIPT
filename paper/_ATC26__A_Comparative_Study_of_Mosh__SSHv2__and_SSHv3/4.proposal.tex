\label{sec:proposal}

This section presents the experimental methodology used to evaluate SSHv2, Mosh, and SSH3 under interactive AI agent terminal workloads. We quantify protocol behavior using terminal events observable at the benchmark client rather than relying only on network-level indicators such as round-trip time or throughput.

\subsection{Measurement Framework}
\label{subsec:measurement-framework}

Figure~\ref{fig:measurement-framework} illustrates the measurement framework. It consists of a benchmark client, a network environment, a remote server, workload generation logic, and a data-analysis process. The benchmark client establishes remote sessions, generates workloads, sends commands or probe characters, observes terminal output, and records timestamps for each measurement event. The same workload suite is executed over SSHv2, Mosh, and SSH3. Traffic then traverses either a controlled LAN environment with tc/netem-based emulation or a VPN-based environment.

On the remote server, sshd, mosh-server, and ssh3-server are installed and listen concurrently. However, only one protocol is active in each experimental run. This design avoids cross-protocol interference during measurement while keeping the server hardware, operating system, account configuration, and workload content consistent across protocols. The controlled LAN environment isolates the effect of configured network impairments, whereas the VPN-based environment captures behavior over a practical overlay deployment path.

\subsection{Experimental Setup and Tools}
\label{subsec:experimental-setup-tools}

The experiments are conducted in two environments: a controlled LAN environment and a VPN-based environment. In the controlled environment, two devices are connected through a LAN, and network impairments are configured on eth0 using Linux tc/netem. In the VPN-based environment, the benchmark client runs on a laptop and connects to the Pi2 server through a VPN tunnel; no tc/netem configuration is applied.

In the controlled environment, the benchmark client is denoted as Pi1, and the remote server is denoted as Pi2. The two devices are connected through the LAN and use eth0 as the measurement interface. In the VPN-based environment, the laptop serves as the benchmark client and connects to the same Pi2 server through the VPN tunnel.

The benchmark client runs Python scripts based on pexpect to establish sessions, issue commands or probes, observe terminal output, and write CSV logs. Each measurement record is associated with protocol, scenario, workload, command or probe type, session ID, sample ID, timestamp information, and success or failure status.

The controlled environment uses three impairment profiles, shown in Table~\ref{tab:network-scenarios}: Low, Medium, and High. These labels indicate increasing network degradation rather than absolute worst-case Internet conditions. Low represents a stable LAN-like setting, Medium introduces noticeable delay, jitter, and packet loss, and High stresses the protocols with lower bandwidth, higher delay, and higher loss.

Because Linux netem applies impairments only to the egress direction, the same configuration is applied on both the client and the server to emulate bidirectional degradation. Thus, when the configured one-way delay is $D$ ms per side, the expected round-trip time is approximately $2D$ ms, excluding jitter and host-processing overhead. The loss values in Table~\ref{tab:network-scenarios} are also per-egress values. If packet loss $p$ is applied in both directions, the approximate bidirectional loss exposure for a request-response exchange is $1-(1-p)^2$.

\subsection{Protocol Implementations and Measurement Semantics}
\label{subsec:protocol-implementations-measurement-semantics}

We evaluate concrete protocol implementations rather than abstract protocol models. SSHv2 is implemented with OpenSSH 9.2p1 and OpenSSL 3.0.17. Mosh is implemented with Mosh 1.4.0 and uses the standard SSH bootstrap procedure before switching to its interactive UDP channel. SSH3 is implemented using francoismichel/ssh3 version 0.1.7 and connects to the /ssh3-term server endpoint.

All protocols use the same remote account, SSH key material, server host, terminal environment, and workload content. SSHv2 is invoked through OpenSSH with pseudo-terminal allocation enabled. Mosh is invoked through the Mosh client using the same SSH bootstrap command, with local input prediction consistently enabled across Mosh trials. SSH3 is invoked through the SSH3 client with the corresponding private key and endpoint configuration. Runtime options, including source IP, identity file, host-key policy, SSH3 mode, and Mosh prediction mode, are recorded in the experiment metadata or run scripts.

Because the evaluated protocols do not expose identical terminal semantics, all metrics are defined using events observable at the benchmark client, including prompt arrival, completion-marker visibility, and character visibility. The interpretation of Mosh measurements follows the protocol-level trade-off discussed in Section~\ref{subsec:protocols}: input latency reflects visible terminal responsiveness, while large-output latency reflects observable terminal completion rather than byte-complete delivery.

\subsection{Application Workloads}
\label{subsec:application-workloads}

The benchmark includes four workloads: command loop, large-output commands, basic interactive editing, and 5-pane interactive editing. These workloads represent common interaction patterns of interactive AI agents operating through a remote terminal: issuing short commands, receiving large outputs, typing interactively, and typing under concurrent redraw pressure. The benchmark uses prompt markers, completion markers, or client-side echo observations to identify the start and end of each measurement.

Each network scenario uses the same repetition structure. For each protocol, workload, and command or probe combination, the experiment opens 10 independent sessions, and each session collects 100 measured samples unless otherwise noted in the dataset metadata.

The workloads are executed using a blocked randomized design. For each scenario, protocol--workload--command combinations are shuffled using a fixed seed of 42 before execution. This reduces order effects caused by cache state, device temperature, run order, or background variation. The seed and shuffle state are stored with the metadata so that the experiment order can be reconstructed.

The command-loop workload consists of four short commands: ls, df -h, ps aux, and grep -n root /etc/passwd. The large-output workload consists of find /, git status, and docker logs, with a per-sample timeout. The basic interactive-editing workload performs 10 warm-up probes, which are excluded from the summary, followed by 100 measured probes in interactive\_shell, vim, and nano. The 5-pane interactive-editing workload uses a tmux session, where pane 0 receives the probe stream and panes 1--4 generate background output and redraw load. Failures, timeouts, and invalid samples are logged instead of being silently discarded.

\subsection{Measurement Metrics}
\label{subsec:measurement-metrics}

All local timestamps are recorded with nanosecond resolution and converted to milliseconds during preprocessing. Since all latency metrics follow the same event-difference structure, we define a general latency metric as

\begin{equation}
L_m = t_{\mathrm{obs}}^{m} - t_{\mathrm{init}}^{m}.
\end{equation}

Here, $m$ denotes the measurement type, $t_{\mathrm{init}}^{m}$ is the client-side timestamp at which the measured action is initiated, and $t_{\mathrm{obs}}^{m}$ is the client-side timestamp at which the corresponding terminal event is observed. The specific meaning of these two timestamps depends on the workload being measured.

For session setup latency, denoted by $L_{\mathrm{setup}}$, $t_{\mathrm{init}}^{m}$ is the time when the benchmark client launches the protocol command, and $t_{\mathrm{obs}}^{m}$ is the time when the first remote shell prompt is observed at the client side. This metric captures the delay before the agent can start issuing commands in a newly established remote session.

For the command-loop workload, the same formulation is reported as command completion latency, denoted by $L_{\mathrm{cmd}}$. In this case, $t_{\mathrm{init}}^{m}$ is the time when the client sends a short command, and $t_{\mathrm{obs}}^{m}$ is the time when the command completion marker is observed on the terminal. This metric captures the latency of short request-response interactions that frequently appear in agent command loops.

For the large-output workload, the same formulation is reported as output completion latency, denoted by $L_{\mathrm{out}}$. Here, $t_{\mathrm{init}}^{m}$ is the time when the large-output command is sent, and $t_{\mathrm{obs}}^{m}$ is the time when the completion marker becomes visible at the client side. The interpretation of this metric follows the protocol semantics described in Section~\ref{subsec:protocol-implementations-measurement-semantics}. In particular, for Mosh, $L_{\mathrm{out}}$ reflects observable terminal completion rather than byte-complete delivery.

For the basic interactive-editing workload, the same formulation is reported as input-to-visible latency, denoted by $L_{\mathrm{input}}$. In this case, $t_{\mathrm{init}}^{m}$ is the time when the probe character is sent, and $t_{\mathrm{obs}}^{m}$ is the time when that character becomes visible at the client side. This metric is measured in interactive shell, vim, and nano contexts.

For the 5-pane interactive-editing workload, the metric is denoted by $L_{\mathrm{input}}^{\mathrm{5pane}}$ and uses the same definition as $L_{\mathrm{input}}$. The difference is that the probe stream is sent to pane 0 of a tmux session, while panes 1--4 generate background output and redraw load. This setting captures input visibility under concurrent terminal activity.

\subsection{Data Collection and Statistical Analysis}
\label{subsec:data-collection-statistical-analysis}

All measurement records are stored at the sample level. Each record includes the protocol, scenario, workload, command or probe type, session ID, sample ID, measured latency, success or failure status, and error information when applicable. Timeouts and samples that do not match the expected marker are logged separately instead of being discarded, so that both latency and protocol robustness are reflected in the analysis.

For each protocol, scenario, and workload combination, we report the number of valid samples $N$, failures, success rate, and latency statistics. The summary includes mean, median, standard deviation, the 95\% confidence interval of the mean, minimum, maximum, 95th percentile, and 99th percentile. The median represents typical behavior, while the 95th and 99th percentiles capture tail latency that an interactive AI agent may experience during long terminal interaction loops.

The results are interpreted according to the protocol semantics defined in Section~\ref{subsec:protocol-implementations-measurement-semantics}. This keeps the analysis aligned with what the agent actually observes at the terminal and avoids attributing unsupported stream semantics to protocols such as Mosh.
