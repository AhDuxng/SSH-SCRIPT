\section{Introduction}
\label{sec:intro}
Remote terminal access remains a basic control plane for servers, cloud instances, edge devices, and experimental testbeds. Traditionally, this interface has been optimized for human operators. Increasingly, however, automated workflows also drive terminals as closed feedback loops: they issue a command, observe the response, and use that observation to decide the next step. Recent agent systems and command-line benchmarks make this interaction pattern visible at the application layer \cite{yao2023react,Yang2024,merrill2026}. This paper studies the transport layer beneath that pattern. We do not run a full AI agent or measure end-to-end task success. Instead, we model an AI-like interactive terminal client with deterministic workloads that isolate what the remote shell delivers to such a client: setup delay, command-completion latency, visible keystroke feedback, and output completeness.

The protocol choice is not straightforward. SSHv2 remains the de facto secure remote shell, but it inherits TCP's ordered byte-stream behavior and address-bound connection model, which can expose head-of-line (HOL) blocking and reconnection costs under impairment or roaming \cite{Scharf2007,Ylonen2006RFC4251}. Mosh takes a different path: it replaces the byte-stream abstraction with UDP-based terminal state synchronization, improving visible responsiveness while allowing intermediate screen states to be superseded \cite{Winstein2012}. SSH3 redesigns secure remote control over HTTP/3 and QUIC, aiming to reduce setup latency and benefit from QUIC stream multiplexing and connection migration \cite{michel2023,Iyengar2021}. These mechanisms suggest clear advantages, but they stress different notions of ``good'' terminal behavior.

For AI-like terminal interaction, the relevant question is therefore not which protocol is universally fastest. A short command loop is sensitive to setup and request-response delay. A large-output command is governed by byte-complete delivery and content correctness. Interactive editing is governed by when characters become visible, even if the underlying protocol does not deliver every intermediate byte. A single aggregate latency number hides these distinctions and can even be misleading: a protocol may look fast because it displays the latest screen quickly while skipping output that an automated client needs to parse.

Existing studies provide valuable but separate views of this design space. Mosh has been evaluated primarily for human-perceived responsiveness under mobile connectivity \cite{Winstein2012}. QUIC studies show that transport benefits depend on workload, network path, and implementation behavior \cite{Langley2017,Kakhki2017}. Agent-oriented benchmarks evaluate task success in command-line environments \cite{Yang2024,merrill2026}, but they do not compare remote-shell protocols under controlled delay, jitter, and loss. This leaves open how SSHv2, Mosh, and SSH3 behave when the same terminal-event workloads are executed over the same network scenarios.

To answer this question, we evaluate SSHv2, Mosh, and SSH3 in two settings: (i) a controlled LAN testbed with tc/netem-based impairment between a benchmark client and a remote server, and (ii) a VPN-based path to the same server. The workload suite contains three classes of terminal events: small-output command loops, large-output fixtures, and interactive editing probes. The results show that protocol performance is strongly workload-dependent. SSH3 achieves the fastest session setup and the lowest small-output command latency. Mosh achieves the lowest visible keystroke latency, especially under impairment, but its screen-synchronization design sacrifices byte-complete large-output delivery. For byte-complete large-output delivery in our implementation and network setup, SSHv2 is faster than SSH3 in the comparable cases.

This paper makes the following contributions:

\begin{itemize}
    \item We present a comparative measurement study of SSHv2, Mosh, and SSH3 for AI-like interactive terminal workloads.
    \item We design a deterministic workload suite that separates setup, small-output commands, large-output delivery, and visible keystroke feedback.
    \item We quantify both latency and output completeness across controlled impairment profiles and a VPN-based environment, showing when protocol design advantages translate into measured benefit and when they do not.
\end{itemize}
