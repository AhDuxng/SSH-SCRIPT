\section{Methodology}
\label{sec:methodology}

This section describes the methodology used to evaluate SSHv2, Mosh, and SSH3 under interactive AI agent terminal workloads. We quantify protocol behavior using terminal events observable at the benchmark client.

\subsection{Experimental Setup and Protocol Implementations}
\label{subsec:measurement-framework}
\label{subsec:protocol-implementations-measurement-semantics}

\begin{figure}[t]
    \centering
    \includegraphics[width=0.9\columnwidth]{figs/workload.pdf}
    \caption{Workload design and client-side measurement events for small-output commands, large-output commands, and interactive editing.}
    \label{fig:setup}
\end{figure}

Figure~\ref{fig:setup} shows the experimental setup used in this study. The setup consists of a benchmark client, a network path, and a remote server. The same workload suite is executed over SSHv2, Mosh, and SSH3, while traffic traverses either a controlled LAN environment with tc/netem-based emulation or a VPN-based environment.

On the remote server, sshd, mosh-server, and ssh3-server are installed and listen concurrently, but only one protocol is active in each experimental run. This avoids cross-protocol interference while keeping the server hardware, operating system, account configuration, terminal environment, and workload content consistent across protocols. The controlled LAN environment isolates the impact of configured impairments, whereas the VPN-based environment captures behavior over a practical overlay deployment path.

In the controlled environment, two Raspberry Pi 4 devices are used: one as the benchmark client and one as the remote server. They are connected directly via Ethernet using eth0, and network impairments are applied with Linux tc/netem. In the VPN-based environment, a laptop running Linux 6.19.11-1-cachyos connects through a VPN tunnel to the same remote server. The benchmark client uses Python/pexpect scripts to establish sessions, issue commands or probes, observe terminal output, and write per-sample CSV logs; the recorded fields are defined in Section~\ref{subsec:data-collection-statistical-analysis}.

The network conditions are summarized in Table~\ref{tab:network-scenarios}. The controlled environment uses three configured impairment profiles: Low, Medium, and High. The VPN-based environment has measured path characteristics collected during the VPN-based experiments rather than configured with tc/netem.

Because netem shapes only egress traffic, we apply the same profile on both hosts to emulate bidirectional degradation; thus a configured one-way delay $D$ yields $\approx 2D$ ms RTT, and per-egress loss $p$ yields $\approx 1-(1-p)^2$ round-trip loss exposure. We evaluate concrete protocol implementations rather than abstract protocol models. SSHv2 is implemented with OpenSSH 9.2 and OpenSSL 3.0.17. Mosh is implemented with Mosh 1.4.0 and uses the standard SSH bootstrap procedure before switching to its interactive UDP channel. SSH3 is implemented using francoismichel/ssh3 implementation and connects to the SSH3 terminal server endpoint \cite{michel2023}.

All protocols use the same remote account, SSH key material, server host, terminal environment, and workload content. SSHv2 is invoked through OpenSSH with pseudo-terminal allocation enabled. Mosh is invoked through the Mosh client using the same SSH bootstrap command, with local input prediction enabled. SSH3 is invoked through the SSH3 client with the corresponding private key and endpoint configuration. Runtime options, including source IP, identity file, host-key policy, SSH3 mode, and Mosh prediction mode, are recorded in the experiment metadata or run scripts.

Because the protocols expose different terminal semantics, all measurements are defined using client-observable events, including prompt arrival, completion-marker visibility, and character visibility. For Mosh, input latency reflects visible terminal responsiveness, whereas large-output latency reflects observable terminal completion.

\subsection{Workload Design, Metrics, and Data Collection}
\label{subsec:application-workloads}
\label{subsec:measurement-metrics}
\label{subsec:data-collection-statistical-analysis}

\begin{figure}[t]
    \centering
    \includegraphics[width=0.9\columnwidth]{figs/setup.pdf}
    \caption{Experimental setup for evaluating SSHv2, Mosh, and SSH3 across controlled LAN and VPN-based environments.}
    \label{fig:workload}
\end{figure}

Figure~\ref{fig:workload} illustrates the workload design and the client-side measurement events used by the benchmark. The benchmark client establishes remote sessions, executes workloads, sends commands or probe characters, observes terminal output, and records timestamps for each measurement event. Prompt markers, completion markers, and client-side echo observations are used to identify measurement boundaries.

The benchmark includes three workloads: W1 for small-output commands, W2 for large-output commands, and W3 for interactive editing. W3 includes both single-pane and 5-pane variants. These workloads represent common terminal interaction patterns of interactive AI agents: issuing commands with small outputs, receiving large outputs, and editing interactively while one or more terminal panes remain active.

\paragraph{W1: Small-output commands.}
Four commands with outputs under 1\,KB: ls, df -h, grep -n root /etc/passwd, git status.

\paragraph{W2: Large-output commands.}
Three commands producing tens of kilobytes: yes $\mid$ head -c N, seq, cat.

\paragraph{W3: Interactive editing.}
A single-pane variant (shell, \texttt{vim}, \texttt{nano}) and a 5-pane variant (\texttt{tmux}: pane~0 receives the probe stream while panes~1-4 generate background redraw load).

For each protocol and workload, the experiment opens 100 independent sessions; each session is established once and then issues 100 command or probe samples. Session setup is therefore measured once per session, and because setup is independent of the workload that follows, we pool these setup measurements across all sessions of a given (protocol, scenario) configuration. A 20\,s timeout is applied to command samples that wait for an observable completion state. Within each scenario, combinations of protocol, workload, and command or probe type are shuffled using a fixed seed before execution. This reduces order effects caused by cache state, device temperature, run order, or background variation. The seed and shuffle state are stored with the metadata so that the experiment order can be reconstructed.

All local timestamps are recorded with nanosecond resolution and converted to milliseconds during preprocessing. Since all latency metrics follow the same event-difference structure, we define a general latency metric as
\begin{equation}
L_m = t_{\mathrm{obs}}^{m} - t_{\mathrm{init}}^{m},
\end{equation}
where $m$ identifies the measurement type, $t_{\mathrm{init}}^{m}$ is the client-side timestamp when the measured action is initiated, and $t_{\mathrm{obs}}^{m}$ is the client-side timestamp when the corresponding terminal event becomes visible. We instantiate $L_m$ for five events: session setup ($L_{\mathrm{setup}}$: launch $\to$ first prompt), command completion ($L_{\mathrm{cmd}}$: W1 send $\to$ completion marker), output completion ($L_{\mathrm{out}}$: W2 send $\to$ completion marker), and input-to-visible latency in the single-pane ($L_{\mathrm{input}}$) and 5-pane ($L_{\mathrm{input}}^{\mathrm{5pane}}$) settings (probe send $\to$ character visible at the client; the 5-pane case observes pane 0 while panes 1-4 generate background redraw load). For Mosh, $L_{\mathrm{out}}$ reflects observable terminal completion rather than byte-complete delivery, and $L_{\mathrm{input}}$ reflects predictive local echo.

\paragraph{Output completeness.}
To quantify how much command output is observed at the client, we compare the received output size of each output-producing sample with a reference output size for the same command. ANSI control sequences are removed before comparison. The reference output size is the median output size from three SSH no-PTY reference runs. The received-output percentage is computed from the received output size and the reference output size, capped at 100\%. If the reference output is empty, the sample is treated as complete. For each protocol, scenario, workload, and command combination, we report the average and minimum received-output percentages over all non-warmup samples.

All measurement records are stored at the sample level. Each record includes the protocol, scenario, workload, command or probe type, session ID, sample ID, measured latency, success or failure status, and error information when applicable. Failures, timeouts, and samples that do not match the expected marker are logged separately rather than discarded. This allows the analysis to capture both latency and protocol robustness. For each protocol, scenario, and workload combination, we report the mean latency. In the High-impairment scenario, we additionally report the p95 latency, which captures the tail of an agent's timeout. For output-producing workloads, we additionally report the average received-output percentage as defined above.
